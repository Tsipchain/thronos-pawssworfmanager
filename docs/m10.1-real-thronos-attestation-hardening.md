# M10.1 — Real Thronos attestation hardening

This milestone hardens the first real Thronos submission path before expanding to generic RPC real execution.

## Hardening rules

- JSON-RPC envelope must be an object with `jsonrpc: "2.0"`.
- RPC responses with `error` are classified as `attestation_rpc_error`.
- RPC `result` must be an object.
- Optional submission `status` must be `accepted` or `submitted`; otherwise rejected.
- `tx_hash` is required and must match `^0x[a-fA-F0-9]{64}$`.
- `attestation_id` is optional; when absent, deterministic id is derived from tx hash prefix.

## Outcome classification

- `submitted`: response validated and accepted into local receipt contract.
- `attestation_rpc_timeout`: transient transport timeout.
- `attestation_rpc_unreachable`: transient transport connectivity failure.
- `attestation_rpc_error`: permanent RPC-layer error.
- `attestation_rpc_malformed_envelope`: permanent malformed JSON-RPC envelope.
- `attestation_rpc_malformed_result`: permanent malformed JSON-RPC result.
- `attestation_invalid_tx_hash`: permanent missing/invalid tx hash.
- `attestation_submission_rejected`: permanent explicit non-accepted status.

## Receipt guarantee/non-guarantee

### Guaranteed now

- `submitted` means the node returned a syntactically valid accepted/submitted result with a valid tx hash.
- Receipt includes tx hash and execute-mode visibility.

### Not guaranteed yet

- inclusion/finality on chain
- signer wallet authorization semantics
- replay protection/nonces policy
- cross-chain payment/runtime integration
