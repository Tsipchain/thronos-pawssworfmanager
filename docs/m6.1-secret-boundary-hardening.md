# M6.1 — Secret-Boundary Hardening

M6.1 hardens provider config and secret handling so forbidden fields cannot leak in current scope.

## Field classification

### Blob provider fields
- public: `provider`, `bucket`, `region`, `endpoint_url`
- sensitive-ref: `access_key_ref`, `secret_key_ref`
- forbidden-raw: `access_key_raw`, `secret_key_raw`

### Attestation provider fields
- public: `rpc_url`, `chain_id`, `contract_address`
- sensitive-ref: `signer_key_ref`
- forbidden-raw: `signer_key_raw`

## Redaction matrix

- public -> report as-is
- sensitive-ref -> report only as `<redacted:set>` when configured
- forbidden-raw -> never report; startup refuses if provided

## Startup refusal rules

Startup rejects:
- plaintext secret env inputs (`BLOB_ACCESS_KEY`, `BLOB_SECRET_KEY`, `ATTESTATION_SIGNER_KEY`)
- contradictory provider settings for disabled backends (`in_memory` blob with provider fields, `fake` attestation with chain fields)
- partial sensitive pairs (`access_key_ref` without `secret_key_ref` and vice versa)
- partial attestation signer pair (`contract_address` without `signer_key_ref` and vice versa)

## Guarantees

- `/v1/config`, `/v1/capabilities`, `/v1/metadata` do not expose raw secrets.
- secret refs are always redacted in reporting.
- forbidden-raw inputs fail closed at startup.

## Non-guarantees

- no real secret manager integration
- no real provider client execution
- no auth/encryption runtime
- no DB/distributed coordination
