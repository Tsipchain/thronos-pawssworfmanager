# M13.1 — Generic RPC execution policy hardening

M13.1 hardens policy/readiness semantics for `rpc_generic` while keeping real execution disabled.

## Policy matrix hardening

- `rpc_generic+dry_run` is allowed for contract-prep behavior.
- `rpc_generic+execute` remains forbidden (fail-closed).

## Readiness vs enabled semantics

`rpc_generic_policy` is now exposed in capabilities/metadata with:

- `selected_backend`
- `policy_pair`
- `policy_allows_pair`
- `execute_forbidden_in_m13_1` (always `true`)
- `readiness` (true only when backend selected + dry_run pair)
- `enabled` (always `false` in M13.1)
- `denial_reason` (`policy_forbids_generic_rpc_execute_in_m13_1`)

## Provider contradiction checks

Backend-specific contradiction rules are now fail-closed:

- `thronos_network` cannot include generic-RPC method fields (`rpc_submit_method`, `rpc_poll_method`)
- `rpc_generic` cannot include Thronos-only `contract_address`

## Guarantees

- explicit policy/readiness contract for `rpc_generic`
- reporting consistency across capabilities and metadata
- no accidental generic RPC execute enablement in this milestone

## Still out of scope

- generic RPC real execution
- wallet auth/signer runtime
- cross-chain payment logic
- encryption runtime
- database/distributed coordination
