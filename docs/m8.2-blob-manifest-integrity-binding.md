# M8.2 — Blob ↔ Manifest Integrity Binding

M8.2 binds blob persistence to deterministic manifest bytes to close the tamper/swap gap before encryption and real attestation.

## Rules

- Blob hash is computed as `sha256(canonical_bytes)` before write.
- `manifest_hash` supplied to orchestrator must match that computed blob hash.
- Integrity binding check runs before persistence side effects; mismatch exits fail-closed.
- `blob_id` is derived deterministically from blob hash (full hex digest).
- Write occurs only if the pre-write hash check passes.
- Post-write verification reads the blob and re-hashes it; mismatch fails hard.

## Receipt semantics

`blob_receipt` now includes:

- `blob_id`
- `blob_hash`
- `verified`
- `status`
- `error_code`
- `failure_class`

New mismatch error:

- `blob_hash_mismatch` (permanent)

## Guarantees

- Same canonical bytes produce same `blob_id` and `blob_hash`.
- Hash mismatch is detected before write.
- Immediate read-back mismatch is detected after write.

## Non-guarantees

- No encryption runtime yet.
- No distributed consistency/locking.
- No blockchain attestation submission yet.
