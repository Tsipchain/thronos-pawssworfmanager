# M10.2 — Thronos attestation lifecycle hardening

This milestone hardens lifecycle semantics after initial real submission boundary.

## Lifecycle states (current scope)

- `submitted_not_finalized`: submission accepted and tx hash validated; finality unknown.
- `submission_rejected`: backend explicitly rejected submission.
- `submission_unknown`: malformed/missing submission evidence; final outcome unknown.
- `submission_failed_retryable`: transport/transient failure, retry may succeed.
- `submission_failed_permanent`: non-retryable failure under current contracts.

## Submitted semantics

`submitted` means:

- JSON-RPC envelope validated
- result validated
- status accepted/submitted
- tx hash validated (`0x` + 64 hex)

`submitted` does **not** mean:

- finality/inclusion proven
- replay/nonce handling complete
- signer runtime assurances

## Receipt consistency

Attestation receipt now includes `lifecycle_state` alongside `status` and `tx_hash`.
When lifecycle is not `submitted_not_finalized`, error responses include `error_code`, `failure_class`, and lifecycle classification.

## Out of scope

- finality polling
- nonce management
- signer runtime
- generic RPC real execution
- wallet auth/cross-chain payments
- encryption runtime / DB / distributed coordination
