# M12.1 — Confirmation proof schema hardening

M12.1 hardens `confirmation_proof` from free-form metadata into a strict Thronos proof contract.

## Strict proof schema (current Thronos scope)

`confirmation_proof` MUST be an object with:

- required fields:
  - `proof_source` (string, must be `"thronos_rpc"`)
  - `proof_kind` (string, must be `"status_attestation"`)
  - `provider_status` (string)
- optional field:
  - `confirmation_id` (string or `null`)

No additional fields are allowed.

## Consistency rules

Proof-bearing reconciliation results are validated fail-closed:

- `confirmation_proof` must be present and typed correctly
- `confirmation_id` (top-level receipt) must equal `confirmation_proof.confirmation_id`
- `provider_status` must match confirmation/finality outcome:
  - `confirmed + finalized` => `provider_status=finalized`
  - `confirmed + not_finalized` => `provider_status=confirmed`
  - `still_pending` => `provider_status in {pending, submitted}`
  - `rejected_or_dropped` => `provider_status in {rejected, dropped}`

Malformed or inconsistent proofs return `invalid_confirmation_proof`.

## What confirmation_proof proves

- The Thronos adapter observed a provider status assertion with constrained schema.
- The receipt outcome is internally consistent with the provider status class.

## What confirmation_proof does NOT prove

- Independent consensus proof beyond provider response.
- Wallet-level signer/nonce replay guarantees.
- Cross-provider or cross-domain finality guarantees.

## Out of scope

- generic RPC real execution
- wallet auth
- cross-chain payment logic
- encryption runtime
- database/distributed coordination
