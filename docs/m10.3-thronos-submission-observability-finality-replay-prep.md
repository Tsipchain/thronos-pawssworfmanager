# M10.3 — Thronos submission observability and finality/replay preparation

This milestone prepares attestation receipts and state contracts for future finality/replay handling, without implementing polling or nonce management.

## Observability additions

Attestation receipts now separate identities:

- `submission_id`: local submission trace identity
- `tx_hash`: on-chain transaction identity (when available)
- `confirmation_id`: reserved for future confirmation/finality identity
- `reconciliation_id`: future polling/reconciliation correlation key
- `confirmation_status`: currently `not_polled`

## Current proof boundary

After `submitted_not_finalized`, the system can prove:

- submission response passed validation
- tx hash format is valid
- lifecycle is explicitly classified

It cannot yet prove:

- chain inclusion/finality
- replay/nonce safety
- signer-side broadcast guarantees beyond provider response

## Future-finality readiness (without polling)

- Receipt contracts already carry confirmation placeholders.
- Lifecycle states are explicit for submission vs unknown/rejected/failure outcomes.
- Reconciliation key is available for future polling workers.

## Out of scope

- finality polling
- nonce management
- generic RPC real execution
- signer runtime / wallet auth
- cross-chain payment logic
- encryption runtime / DB / distributed coordination
