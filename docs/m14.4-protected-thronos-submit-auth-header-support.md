# M14.4 — Protected Thronos submit auth-header support

This milestone prepares `thronos_network` submit calls for protected `/api/tx_submit` style routes by adding configurable auth-header injection with ref-based secret handling.

## Added provider config contract

- `ATTESTATION_AUTH_HEADER_NAME` (public)
- `ATTESTATION_AUTH_HEADER_REF` (sensitive ref, redacted in outputs)
- `ATTESTATION_AUTH_HEADER_PREFIX` (public, optional; e.g. `Bearer`)

If one of `name/ref` is set without the other, startup fails (`incomplete_attestation_auth_header_pair`).

Plaintext secret env input is forbidden:

- `ATTESTATION_AUTH_HEADER_VALUE`

## Submit header injection behavior

For `RealThronosAttestationAdapter` submit calls:

- if no auth header config is provided, request behavior is unchanged
- if configured:
  - header key = `ATTESTATION_AUTH_HEADER_NAME`
  - header value = resolved value from `ATTESTATION_AUTH_HEADER_REF`
    - if ref is `env://VAR_NAME`, runtime resolves from environment variable `VAR_NAME`
    - if ref is not `env://...`, it is used literally
  - if prefix is set, value becomes `<prefix> <ref>`

Examples:

- `Authorization` + `Bearer` + `ref://token` => `Authorization: Bearer ref://token`
- `X-Internal-Key` + (no prefix) + `ref://k` => `X-Internal-Key: ref://k`
- `Authorization` + `Bearer` + `env://THRONOS_SUBMIT_TOKEN` => `Authorization: Bearer <resolved env value>`

## Unchanged

- generic RPC path is unchanged
- receipt/error contracts remain stable
- upstream protected-route contract details (exact header names/values, allowlist policy) remain external requirements
